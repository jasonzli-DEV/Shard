# Shard v2 — Config-in-DB, Vercel, Access Control, Resilience

> Execute with superpowers:subagent-driven-development. Each phase = one Sonnet implementer, TDD, then verify. Builds on the shipped v1 (backend 316 / frontend 84 / e2e 8 green, deployed to pi.local).

**Goal:** Make Shard portable (deployable on Vercel as well as Docker/Pi), restart-free, with an admin-controlled access model and graceful handling when the starter (metadata) cluster fills or fails.

## Global Constraints
- Keep all existing tests green; add tests for new behavior (TDD).
- Only `STARTER_MONGODB_URI` (+ optional `CRON_SECRET`, `COOKIE_SECURE`, `PORT`, `SERVERLESS`) may be env vars. ALL other config (OAuth creds, public URL, allowed origins, JWT secret) moves into the starter DB.
- Must still run as Docker on the Pi (background loops) AND on Vercel (serverless + cron). Gate in-process loops behind `SERVERLESS!=1`.
- No stubs/TODOs in shipped code. TypeScript strict.

## Phase A — Config in the database (foundation)
Replaces `.env`-writing setup with DB-stored config. Fixes restart-free setup and is the basis for Vercel (read-only FS).

- New `Config` model in the starter DB: a singleton doc `{ _id:'singleton', googleClientId?, googleClientSecret?, githubClientId?, githubClientSecret?, publicUrl?, allowedOrigins?, jwtSecret, updatedAt }`. Secrets stored plaintext (consistent with v1's intentional model).
- `backend/src/config/configService.ts`: `loadConfig()` (reads the singleton from DB, caches in memory, refreshes on write), `getConfig()` (sync accessor over cache, falls back to `process.env` for each key so existing env deploys keep working), `saveConfig(patch)` (upserts + refreshes cache + auto-generates jwtSecret if missing). Bootstraps before routes need it.
- Rewrite `routes/setup.ts`: `status` derives setupRequired from the Config doc (has starter URI via env + ≥1 OAuth provider in Config). `configure` writes to Config (NOT `.env`), refreshes cache, re-inits passport, calls `connectRuntime`. Drop the `.env`-file write path (and `SETUP_ENV_FILE_PATH`, the config volume).
- `passport.ts`, CORS in `app.ts`, JWT signing — read from `getConfig()` not `process.env` directly.
- Tests: config save/load/cache + env fallback; setup status from DB; configure persists to Config.

## Phase B — Serverless / Vercel plumbing
- Mongoose cached-connection helper (reuse the connection across warm invocations; don't reconnect per request). Apply to starter + default connections.
- Refactor `scheduler.ts`: extract loop bodies into callable functions `runStorageCheckAllUsers()` and `runDecommissionSweep()`. `startStorageLoops()` (persistent host) calls them on intervals, guarded by `SERVERLESS!=1`.
- Cron endpoints: `POST /api/cron/storage-check`, `POST /api/cron/decommission`, authorized by `CRON_SECRET` (header `x-cron-secret` or Vercel's `Authorization: Bearer`). Call the same functions. Drop the per-minute keepalive on serverless (document why: M0 pauses only after ~60 days idle).
- `vercel.json` (frontend static build + backend as a Node serverless function wrapping the express app via `api/index.ts`; cron schedules). A serverless entrypoint that does NOT `app.listen`.
- Keep Docker/Pi path intact. Docs: README "Deploy on Vercel" + env vars.
- Tests: cron endpoint auth (401 without secret), cron triggers the maintenance fn; loops gated by SERVERLESS.

## Phase C — Admin-controlled access model
Answers "why only invited users": currently any OAuth user can sign in. Make access admin-controlled.
- User model: add `status: 'active' | 'pending'`. First user ever → `role:'admin', status:'active'`. Subsequent first-time sign-ins → `status:'pending'` (created but no access).
- `requireAuth`: a `pending` user is authenticated-but-unauthorized — reject protected routes with 403 `{reason:'pending_approval'}`. Pending users can hit `/api/me` (to see their status) and logout only.
- Admin endpoints: `GET /api/admin/users`, `POST /api/admin/users/:id/approve`, `POST /api/admin/users/:id/deny` (deny deletes or marks denied), `POST /api/admin/users/:id/role`. Admin-only.
- Optional invite: `POST /api/admin/invites` {email} → pre-approves that email so they're `active` on first sign-in. (Keep simple: email allowlist in Config or an `Invite` collection.)
- Frontend: pending users see an "Awaiting approval" screen instead of the Drive; admin gets a Users panel in Settings (approve/deny/role, list pending+active). Login subtitle accurate ("Approved accounts only" or similar).
- Tests: first user admin+active; second user pending; pending blocked from files with 403 pending_approval; admin approves → active gains access; non-admin can't reach admin routes.

## Phase D — Starter cluster resilience
Ensure metadata-cluster exhaustion/failure degrades gracefully.
- `starterHealth` service: periodic (loop + cron) `dbStats` on the starter DB; store usage; expose via `GET /api/storage` (and a dedicated admin field). Warn threshold (e.g. 80% of a configurable `STARTER_CAPACITY_BYTES`, default 512MB).
- Graceful write failures: a central handler maps Mongo quota/space/connection errors (e.g. code 8000/`over your space quota`, ECONNREFUSED, pool errors) on metadata writes to a clean 507/503 with an actionable message ("Metadata store is full or unreachable — the admin must upgrade the starter cluster"). The process must NOT crash; unhandled rejections/exceptions are logged, not fatal (already partly via error middleware — verify uploads/operations return clean errors).
- Dashboard: starter-cluster usage bar + warning banner for admins when near capacity; reads succeed read-only even when writes would fail.
- Tests: starter usage reported; simulated write-quota error returns 507 not a crash; warning surfaced past threshold.
